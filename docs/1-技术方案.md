# Personal Planner — 个人规划与每日复盘系统 技术方案

> 一个用于"中长期目标 + 每日任务 + 每日复盘 + 个人画像动态更新"的个人工具。

---

## 一、项目定位

**核心问题**：每天忙忙碌碌，但说不清"长期要做什么 → 今天要做什么 → 今天做了什么"，学了东西不知道现在到什么水平。

**解决思路**：把"目标管理 + 任务执行 + 每日复盘 + 个人画像动态更新"打通到同一个工具里。
- 中长期：季度/年度目标（OKR 风格）
- 每日任务与执行追踪
- 每日复盘（记录学习内容）
- **个人画像动态更新**（AI 自动分析学习记录，更新技能评级 + 生成 MD 画像文档）

**目标用户**：自己（用项目管理工具管理自己的技术成长）。

## 二、功能模块

### 2.1 目标管理（Goal）
- 多层级目标：年度 → 季度 → 月度 → 周
- 字段：标题、描述、起止时间、状态（active/done/archived）、进度（0-100%）
- 关键结果（Key Result）量化指标
- 目标 ↔ 任务 多对多关联（带 contribution 贡献度）

### 2.2 任务管理（Task）
- **今日视图**（首页）：今天待办清单
- **全部任务**：跨天总任务列表，按状态/优先级/截止时间筛选
- 字段：标题、描述、状态（todo/doing/done/abandoned）、优先级（low/medium/high/urgent）、预计耗时、实际耗时、计划日期、截止日期
- 子任务拆解

### 2.3 每日复盘（Daily Review）
- 一天一篇（review_date UNIQUE）
- 字段：今日完成、今日未完成、今日收获、今日改进、心情打分
- 自动从已完成/未完成任务带入

### 2.4 标签（Tag）
- 自定义标签（user 隔离）
- 任务 ↔ 标签 多对多

### 2.5 学习记录（Learning Record）
- 每次学习一条记录：标题、内容、时长、标签（JSON）、来源
- 是 AI 更新画像的数据源

### 2.6 数据分析（Analytics）
- 热力图（每日完成度）
- 按标签时间分布
- 完成率趋势

### 2.7 个人画像（Skill Profile）— 核心特色模块

**数据内容**：
- 技能分类（一级分类：Java后端/前端/AI/基础设施/软技能）
- 每个技能：名称、评级（S/A/B/C/D）、关键词、备注
- 每次更新记录变更历史（升/降/新增/手动/AI）

**三种更新方式**：
1. **手动编辑**：直接增删改技能
2. **AI 增量更新**（`/api/profile/ai-update`）：基于最近 N 天学习记录，LLM 返回评级变更
3. **AI 完整画像生成**（`/api/profile/ai-generate-md`）：上传简历/技术总结 MD → LLM 生成结构化画像文档，保存到数据库

## 四、核心数据模型

### 4.1 表结构（12 张主表）

| # | 表名 | 说明 |
|---|------|------|
| 1 | `sys_user` | 用户表 |
| 2 | `goal` | 目标（自关联 parent_id，level: year/quarter/month/week） |
| 3 | `key_result` | 关键结果（goal_id） |
| 4 | `task` | 任务（status/priority ENUM） |
| 5 | `sub_task` | 子任务 |
| 6 | `tag` | 标签（user 隔离） |
| 7 | `task_tag` | 任务-标签关联 |
| 8 | `goal_task` | 目标-任务关联（contribution 贡献度） |
| 9 | `daily_review` | 每日复盘（review_date UNIQUE） |
| 10 | `skill_profile` | 技能画像（UNIQUE(user_id, category, skill_name)，level: S/A/B/C/D） |
| 11 | `profile_change_history` | 画像变更历史（change_type: increase/decrease/new/manual/ai） |
| 12 | `learning_record` | 学习记录（tags JSON 字段） |
| 13 | `profile_document` | **画像文档**（保存 AI 生成 / 用户编辑后的 MD 全文） |

### 4.2 关键设计决策

**为什么 task 表有 `planned_date` 又有 `due_at`？**
- `planned_date`：今天做哪件事（默认今日）
- `due_at`：必须什么时候完成（截止时间，可能跨天）
- 解耦：今天做但本周结束前完成，是常见场景

**为什么 goal 和 task 是多对多？**
- 一个任务可能推进多个目标
- 引入 `goal_task` 中间表，带 `contribution` 字段

**为什么 `profile_document` 独立成表？**
- 完整画像可能数千字，MEDIUMTEXT 存储
- 包含 version 字段（每次更新 +1），方便追踪
- 与 `skill_profile`（结构化评级）解耦：用户可手动编辑 MD，AI 重新生成会覆盖

## 六、AI 画像功能详解

### 6.1 AI 增量更新（`/ai-update`）

**原理**：把"最近 N 天的学习记录 + 当前技能画像"发给 LLM，LLM 返回每项技能的评级变更建议。

```
学习记录 (N 天)  ──┐
                   ├──→ LLM ──→ JSON {updates: [...], summary}
当前技能画像  ──────┘                ↓
                              normalizeLevel(B+ → A)
                              写库 + 记 history
```

**Prompt 关键约束**：
- "必须有学习记录支撑才更新"
- "不要过度提升"
- LLM 返回 B+/A- 等过渡评级时，归一化到 S/A/B/C/D 枚举

### 6.2 AI 完整画像生成（`/ai-generate-md`）— 核心特色

**两种输入**：
1. 上传/粘贴 Markdown 文档（简历、技术总结、项目经历）
2. 可选：参考最近 30 天学习记录

**输出结构**（强制 6 个章节）：

```markdown
# 个人技术画像

## 1. 总体定位
（一句话总结技术方向与角色定位）

## 2. 核心技能（按一级分类）
### Java后端
- **Spring Boot (A)**：xxx
### 前端
- **Vue (B)**：xxx
### 中间件
### AI
（按实际内容省略或添加）

## 3. 技术亮点
- （3-5 条）

## 4. 经验与项目沉淀
- （3-6 条，保留关键数据：耗时/性能提升/代码量等）

## 5. 待补充的技术（按优先级 P0/P1/P2）
### P0 — 必须补齐
### P1 — 应该补齐
### P2 — 了解即可

## 6. 学习建议
- 短期（1 个月）
- 中期（3 个月）
- 长期（6-12 个月）

---
**评级说明**: S=精通 | A=熟练 | B=掌握 | C=了解 | D=入门
```

**Prompt 核心原则**：
1. **优先使用上传文档**的事实
2. **用户自评**（熟练/掌握/了解）映射到 S/A/B/C/D
3. 不编造用户没提到的内容
4. 吸收 P0/P1/P2 优先级和能力矩阵等结构

**关键参数**：
- 上传文档截断长度：12000 字符
- LLM max_tokens：4096
- LLM HTTP 读取超时：120 秒

### 6.3 画像文档的编辑能力

`profile_document` 表保存每次生成的 MD 全文（含 version 字段）。前端提供：

| 模式 | 入口 | 说明 |
|------|------|------|
| 预览 | 默认 | markdown-it 渲染为 HTML，结构化样式 |
| 编辑 | 顶栏「编辑」按钮 | 切换为 textarea（深色代码风格），用户可自由修改 |
| 重新生成 | 上传新文档后点「AI 重新生成画像」 | LLM 覆盖 |
| 保存编辑 | 编辑模式点「保存」 | PUT /document，version +1 |

### 6.4 LLM 集成（MiniMax M3）

```yaml
# application.yml
minimax:
  api-key: sk-cp-...
  base-url: https://api.minimaxi.com/anthropic
  model: MiniMax-M3
```

**协议**：Anthropic 兼容 API
- Endpoint: `POST {base-url}/v1/messages?model={model}`
- Headers: `x-api-key`, `anthropic-version: 2023-06-01`
- Request: `{messages: [{role: user, content: prompt}], max_tokens: 4096}`
- Response: `content[].text` 提取文本

## 八、安全与会话

### 8.1 JWT 配置
```yaml
jwt:
  secret: PersonalPlanner2026SecretKeyForJwtTokenGenerationAndValidation123456
  access-expiration: 86400000      # 1 天（86400 秒）
  refresh-expiration: 604800000    # 7 天
```

### 8.2 登录流程
1. 前端 POST `/api/auth/login` → 返回 `accessToken` + `refreshToken`
2. 存 `localStorage` + Pinia state
3. 所有业务请求 header 注入 Bearer token
4. 后端 `JwtAuthFilter` 解析 token → 设置 `SecurityContext`
5. token 过期 → 401 → 前端弹窗 → 跳转登录

### 8.3 角色与权限
- 单用户系统（仅自己用），无复杂 RBAC
- 所有 Controller 通过 `getUserId(username)` 提取用户身份（当前临时硬编码为 1L）
- TODO：改为从 `SecurityContext` 解析 userId

## 九、数据流：AI 生成画像的完整调用链

```
用户上传 personal-tech-profile.md
  ↓
前端读取 FileReader → mdContent.value
  ↓
点「AI 生成画像」
  ↓
POST /api/profile/ai-generate-md
  {mdContent: "...", includeLearning: true}
  ↓
Controller → saveOrUpdateProfileDocument(userId, req)
  ↓
Service: buildMdProfilePrompt(profile, records, sourceMd)
  - 截断 sourceMd 到 12000 字符
  - 拼接：⚠️ 重要原则 + 用户文档 + 系统技能 + 学习记录 + 输出模板
  ↓
callLlmForMd(prompt) → RestTemplate POST minimax API
  - connect 15s, read 120s
  - x-api-key + anthropic-version headers
  ↓
LLM 返回 markdown
  ↓
extractSuggestedSkills(md) → 解析「## 5. 待补充的技术」章节
  ↓
保存到 profile_document 表（version +1）
  ↓
返回 {markdown, suggestedSkills}
  ↓
前端：渲染预览（markdown-it）+ 展示「待补充的技术」标签
  ↓
用户可点「编辑」→ 改后 PUT /document → version 再 +1
```

## 十一、关键风险与对策

| 风险 | 对策 |
|------|------|
| LLM 超时（30s 不够） | RestTemplate read timeout 设为 120s |
| LLM 返回非法枚举值（B+） | normalizeLevel 归一化函数 |
| 上传 MD 过大（>12K 字符） | 截断 + 提示「文档已截断」 |
| Token 过期导致用户困惑 | 401 拦截器弹窗 + 引导 |
| 数据库 nickname 中文乱码 | 表用 utf8mb4，init.sql 显式指定 |
| 单用户系统，未来多用户 | userId 提取待从 SecurityContext 替换硬编码 |
